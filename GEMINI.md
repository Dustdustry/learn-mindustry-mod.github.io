📝 Mindustry 教程文档编写规范

  为了保证本教程的连贯性、专业性与阅读体验，请未来的文档贡献者在编写或修改章节时，遵循以下风格与规范：

  一、 基调与人称：亦师亦友，保持温度
   1. 使用对话式人称：多使用“我们（进行某项操作）”、“你（将会学到）”来进行表述，拉近与读者的距离。保持耐心、鼓励的导师口吻，避免生硬冷漠的教科书式说教。
   2. 开篇引语：鼓励在每个核心章节的标题下方，引用一句简短的名言或核心理念（如 > ***“万事开头难”***），为硬核的技术文档增添人文气息与阅读趣味。

  二、 内容组织：务实求真，化繁为简
   1. 明确前置要求：在引入新阶段或高阶内容（如OpenGL、AI辅助编程）时，必须在开头清晰列出“学习本章需要具备的基础”，不避讳难点，对初学者负责。
   2. 场景化解释概念：在解释复杂的底层概念（如状态机、渲染管线）时，不要照搬百科定义的晦涩术语。应直接结合 Mindustry 和 Arc
      框架的实际应用场景，告诉读者“它在这里是怎么工作的”、“我们应该如何调用它”。
   3. 主张最佳实践：在存在多种实现方式时，应明确给出推荐方案及其优缺点（例如针对不同人群推荐 JSON 或
      Java），并强调游戏开发中的代码安全与可维护性（如“非必要不使用原始GL工作流”）。

  三、 排版与格式：视觉分层，多维对照
   1. 善用多语言代码块：Mindustry 模组开发支持多种语言。在提供代码示例时，尽可能使用 code-group 提供 Java / Kotlin / JavaScript / JSON
      的多维对照，满足不同技术栈读者的需求。
   2. 合理使用提示组件：避免大段枯燥的纯文字。请熟练使用 Markdown 的提示框语法进行视觉分层：
      - ::: info：用于补充专有名词解释（如“全限定名”）或背景知识。
      - ::: tip：用于提供最佳实践、技巧提示或常见问题解答。
      - ::: warning / 注意：用于强调易错点或版本兼容性问题。
   3. 结构递进：遵循“总述背景 -> 核心概念提取 -> 代码实践演示 -> 效果验证”的逻辑脉络，确保初学者的学习曲线平滑过渡。
